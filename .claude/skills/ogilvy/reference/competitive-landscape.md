# Competitive Landscape

## Market Category

**Design System Platforms** — tools that help teams build, maintain, and enforce design systems. Preset operates at the intersection of design system management and AI-assisted development.

## Competitors

### Knapsack

**Positioning:** Enterprise design system platform for complex organizations.

**Strengths:**
- Deep enterprise features (governance, analytics, multi-brand)
- Strong Figma integration
- Established customer base

**Weaknesses:**
- Complex setup and onboarding
- Enterprise pricing excludes smaller teams
- No AI-native capabilities
- Documentation-focused, not enforcement-focused

**Our angle:** Knapsack manages design systems for humans. Preset makes design systems work for AI too. Different era, different problem.

### Specify

**Positioning:** Design token management and distribution.

**Strengths:**
- Strong token pipeline (Figma > code)
- Multiple output formats
- Good developer experience

**Weaknesses:**
- Token-only scope (no patterns, no presets, no enforcement)
- No AI integration
- Pipeline approach (build-time, not runtime)
- No compliance or health monitoring

**Our angle:** Specify moves tokens from A to B. Preset makes your entire design system executable and AI-accessible. Tokens are table stakes — enforcement is the game.

### Supernova

**Positioning:** Design system documentation and code generation.

**Strengths:**
- Beautiful documentation output
- Code generation from Figma
- Visual editing experience

**Weaknesses:**
- Documentation-first (docs get ignored)
- No runtime enforcement
- No AI tool integration
- Generated code is static, not contextual

**Our angle:** Documentation tells developers what to do. Preset tells AI tools what to do. One gets ignored; the other is enforced automatically.

### Zeroheight

**Positioning:** Design system documentation platform.

**Strengths:**
- Clean documentation experience
- Good Figma/code integration
- Established in the market

**Weaknesses:**
- Pure documentation (no enforcement)
- No AI capabilities
- Static output
- Relies on developers reading docs

**Our angle:** Zeroheight creates beautiful docs that developers bookmark and forget. Preset creates executable context that AI tools use automatically.

### Style Dictionary (Open Source)

**Positioning:** Build-time design token transformer.

**Strengths:**
- Free and open source
- Flexible token transformation
- Large community
- Well-understood pattern

**Weaknesses:**
- Build-time only (no runtime)
- Token-only scope
- No AI integration
- Requires manual pipeline setup
- No enforcement or compliance

**Our angle:** Style Dictionary is infrastructure. Preset is a platform. We can consume Style Dictionary tokens and make them AI-accessible. Complementary, not competitive.

## How Preset Differentiates

### 1. AI-Native (MCP Integration)
No competitor offers real-time AI tool integration via MCP. This is not a feature we added — it is the architectural foundation. Design systems become runtime context for AI, not static documentation.

### 2. Executable Enforcement
Presets are not documentation — they are executable specifications. When AI generates a component, the preset defines exactly what tokens, patterns, and constraints apply. Compliance is automatic, not aspirational.

### 3. Museum Design Philosophy
The platform recedes; the work shines. This is not just aesthetics — it is a product philosophy that makes the tool disappear into the workflow. Competitors build tools that demand attention. We build tools that amplify intent.

### 4. Living System, Not Static Output
Design systems are living artifacts. Preset treats them that way — with health monitoring, drift detection, and continuous enforcement. Competitors snapshot; we stream.

## Messaging Guidelines for Competitive Contexts

- **Never attack competitors by name** in public-facing copy
- **Acknowledge the category** — "design system tools exist, but they were built for a different era"
- **Differentiate on the problem** — "the problem isn't managing tokens; it's making AI respect them"
- **Use proof, not claims** — show the before/after, not the superlative
- **Respect incumbents** — they built the category; we are evolving it
